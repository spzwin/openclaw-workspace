## 四、标准作战流程（6 步循环）

```
第1步 → 第2步 → 第3步 → 第4步 → 第5步 → 第6步 → 回到第1步
扫描    评估    占坑    开发    交付    追踪
```

---

### 第 1 步：扫描机会

**频率：** 每天至少扫一次

**扫描平台：**

| 平台 | 类型 | 优先级 |
|------|------|--------|
| [Superteam Earn](https://earn.superteam.fun) | Web3 任务/Hackathon | ⭐⭐⭐ |
| GitHub（bounty 标签） | 技术 Issue 悬赏 | ⭐⭐⭐ |
| [Gitcoin](https://gitcoin.co) | 开源开发赏金 | ⭐⭐ |
| [DevPost](https://devpost.com) | Hackathon | ⭐⭐ |
| Upwork/Fiverr（固定价格） | 自由职业 | ⭐ |

**筛选条件：**
- ✅ 支付方式：USD / USDC / USDT / ETH / BTC
- ✅ 截止日期：3天内可完成
- ✅ 奖金范围：$20 ~ $500（初期聚焦小单）
- ✅ 技术匹配：Python / JS / TypeScript / 内容创作
- ❌ 拒绝：山寨币支付、支付机制不明确、>7天项目

---

### 第 2 步：ROI 评估

**评估公式：**
```
净利润 = 预估奖金 - Token成本 × 1.5
通过条件：净利润 > 0 且 利润率 > 50%
```

**评估矩阵：**

| 维度 | 打分标准 |
|------|----------|
| 奖金金额 | $20~$100 优先（稳拿）；$100+ 要评估竞争度 |
| 复杂度 | 低(1-2h) > 中(4-8h) > 高(>8h) |
| 竞争度 | 评论少、截止近的优先 |
| 验收清晰度 | 有明确 spec > 模糊需求 |
| 任务类型 | 技术实现 ≈ 内容创作 > 纯设计 |

**快速判断口诀：**
> 奖金清晰、周期短、竞争少 → 直接上
> 奖金模糊、周期长、要投钱 → 果断 pass

---

### 第 3 步：领地声明（占坑）

**为什么要占坑：** 防止其他人先提交，让评审知道你在做

**GitHub Issue 占坑模板：**
```
I'm looking into this and plan to submit a PR shortly.
I have a working approach and will have a solution ready within [X hours/days].
```

**Superteam / 其他平台：** 直接开始做，平台本身有时间戳记录

---

### 第 4 步：方案开发

**技术类任务交付物：**
- `main.py` / `index.ts` — 主程序（功能完整、有注释）
- `test_*.py` / `*.test.ts` — 测试套件（覆盖核心场景）
- `README.md` — 使用说明（清晰、有示例）
- `SOLUTION.md` — 技术方案说明（给评审看）

**内容类任务交付物：**
- 符合平台格式的内容（Thread / 文章 / 视频脚本）
- 数据支撑（截图、链接、来源）
- 发布证明（链接 + 截图）

**质量红线：**
- 代码必须能跑通
- 内容必须符合任务要求的字数/格式
- 不交不完整的东西

## 五、任务类型快速参考

### A. GitHub 技术悬赏
```
发现 Issue → ROI评估 → 评论占坑 → 写代码 → 提交PR（含钱包地址）→ 等待合并+打款
```

### B. Superteam Earn Bounty
```
浏览 earn.superteam.fun → 筛选 Bounty 类型 → 评估ROI → 做内容/代码 → 填写提交表单 → 等待评审
```

### C. Hackathon
```
报名 → 评估赛道（选 AI / Solana / 工具类）→ 快速开发 MVP → 部署线上 → 提交作品 → 等待评审
```

### D. 内容任务（Thread / 文章）
```
理解要求 → 研究主题 → 写内容 → 发布到平台 → 提交截图证明 → 等待打款
```

## 六、效率技巧

1. **先占坑，再开发** — 评论速度比开发速度更关键
2. **模板复用** — PR 模板、README 模板保存好，改改就能用
3. **小单先跑** — $20~$100 的小单先积累成功记录，再冲大单
4. **并行推进** — 同时跟进 2-3 个不同阶段的任务，效率最大化
5. **截止日期倒排** — 优先完成即将截止的任务
6. **失败也记录** — 没中选的任务记录原因，下次优化
