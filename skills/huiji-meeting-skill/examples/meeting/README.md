# 会议模块 — 使用说明

## 什么时候使用

- 用户要查会议列表、搜会议、看进行中的会议时 → 使用 meeting 模块（会议列表、搜索）。
- 用户要「实时总结」「最近 N 分钟讲了什么」「有没有提到 XXX」时 → 使用 meeting 模块（分片转写 + 实时问答）。
- 用户要「总结这场会议」「生成报告」「列待办」「多场综合总结」时 → 使用 meeting 模块（总结/待办/聚合），并配合文末提示词。

## 会议总结报告的约束与模版

用户说「帮我总结这场会议」「生成会议报告」时，生成的报告**必须**遵守以下约定：

- **约束**：报告语言与会议内容一致；严格基于原文、禁止虚构；内容过少（少于 100 字）时只提示「无法生成有效的会议报告」；输出为 Markdown。
- **模版结构**：① 标题 `# [核心主题][会议类型]`（类型限：沟通会/讨论会/培训会/评审会/宣导会/面试/汇报会/决策会/启动会/复盘会）；② 100–300 字概述（无小标题）；③ 详细报告（禁止「会议性质分析」等元描述）；④ 待办事项（仅当存在明确共识且可执行的待办时输出表格，否则不输出该节）。
- 详细说明与完整提示词见 SKILL.md「三、单场会议总结」与文末「会议报告生成提示词」。

## 待办提取的约束与模版

用户说「列一下待办」「有哪些要做的」时，输出遵守：

- **约束**：严格基于原文、禁止虚构负责人/时间；宁少勿多；无待办时禁止输出「该会议暂无待办事项」，可输出一句中性说明如「经分析，本次会议未形成需列出的待办事项。」
- **模版**：有待办时逐条为 `- [负责人] 在 [时间] 前完成 [任务描述]`；无待办时仅一句中性说明。识别标准与报告中的待办一致（明确共识 + 可执行动作 + 明确任务目标）。
- 完整提示词见 SKILL.md「四、待办事项提取」与文末「会议待办事项提取提示词」。

## 鉴权预检

1. 按 `../../common/auth.md` 与 `../../openapi/common/appkey.md` 获取 AppKey（环境变量 `XG_BIZ_API_KEY` 或上下文传入）。
2. 慧记接口在请求头携带 `appKey` 即可。

## 标准流程

1. 鉴权预检（见上）。
2. 根据意图调用对应接口：列表/搜索见 `../../openapi/meeting/chatListByPage.md`，转写与实时问答见 `../../openapi/meeting/splitRecordList.md`。
3. 按 SKILL.md 中的输出格式与提示词生成列表/总结/待办，对用户只输出结论与必要链接，不暴露鉴权信息。

## 脚本与入口

- 根目录 `main.py`：`list_meetings`、`in_progress_meetings`、`understand_query`、`search`、`answer_question`、`summarize_meeting`、`extract_todos`。
