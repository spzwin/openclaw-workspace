# 定时器轮询 vs 事件驱动 - 架构对比

## 核心区别

| 维度 | 定时器轮询（旧） | 事件驱动（新） |
|------|-----------------|---------------|
| **触发方式** | 定时器每秒/分钟检查 | Agent 主动通知 |
| **响应延迟** | 最高 1 分钟（取决于轮询间隔） | 实时（<1 秒） |
| **资源消耗** | 持续轮询，浪费 CPU | 按需触发，零空闲消耗 |
| **实现复杂度** | 需要定时器、状态机、锁 | 简单的事件监听 |
| **可扩展性** | 轮询间隔成为瓶颈 | 天然支持高并发 |
| **调试难度** | 时序问题难复现 | 事件日志清晰可追溯 |

## 工作流程对比

### 定时器轮询模式

```
┌─────────────────────────────────────────────────────────────┐
│                     定时器（每 5 秒）                        │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
              ┌─────────────────────────┐
              │   检查所有项目进度       │
              │   for project in projects:│
              │     read project.json    │
              │     check status         │
              └─────────────────────────┘
                            │
                            ▼
              ┌─────────────────────────┐
              │   发现有任务完成？       │
              │   if task.completed:     │
              │     find_next_task()     │
              └─────────────────────────┘
                            │
                            ▼
              ┌─────────────────────────┐
              │   派发下一个任务         │
              │   dispatch(next_task)    │
              └─────────────────────────┘
```

**问题：**
1. 即使没有任务完成，也要每 5 秒检查一次
2. 项目多的时候，轮询成本很高
3. 任务完成后最多等 5 秒才能被处理
4. 轮询间隔短 → 资源浪费；间隔长 → 响应慢

### 事件驱动模式

```
┌─────────────────────────────────────────────────────────────┐
│                     Agent 完成任务                          │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
              ┌─────────────────────────┐
              │   主动提交通知           │
              │   send_notification()    │
              └─────────────────────────┘
                            │
                            ▼
              ┌─────────────────────────┐
              │   调度中心立即处理       │
              │   handle_notification()  │
              └─────────────────────────┘
                            │
                            ▼
              ┌─────────────────────────┐
              │   更新进度 + 派发下一步  │
              │   update_and_dispatch()  │
              └─────────────────────────┘
```

**优势：**
1. 没有任务时完全不消耗资源
2. 任务完成后立即处理（实时）
3. 架构简单，易于理解和维护
4. 天然支持并发（多个 Agent 同时提交通知）

## 代码对比

### 定时器轮询实现

```typescript
// 旧模式：定时器轮询
class PollingScheduler {
  private interval: NodeJS.Timer;
  
  start() {
    // 每 5 秒轮询一次
    this.interval = setInterval(async () => {
      await this.checkAllProjects();
    }, 5000);
  }
  
  private async checkAllProjects() {
    const projects = await this.getAllProjects();
    
    for (const project of projects) {
      if (project.status !== 'in_progress') continue;
      
      // 读取每个项目的进度
      const progress = await this.readProjectProgress(project.id);
      
      // 检查是否有新完成的任务
      const completedTasks = this.findNewCompletedTasks(progress);
      
      for (const task of completedTasks) {
        // 派发下一个任务
        const nextTask = await this.findNextTask(project.id, task.id);
        if (nextTask) {
          await this.dispatchTask(nextTask);
        }
      }
    }
  }
  
  stop() {
    clearInterval(this.interval);
  }
}

// 问题：
// 1. 即使没有项目在执行，也要轮询
// 2. 项目多的时候，一次轮询可能要几秒
// 3. 轮询间隔难以调优：短了浪费，慢了延迟高
```

### 事件驱动实现

```typescript
// 新模式：事件驱动
class EventDrivenScheduler {
  async handleNotification(notification: Notification) {
    // 只在收到通知时才处理
    const { projectId, taskId } = notification;
    
    // 1. 更新进度
    await this.updateProjectProgress(projectId, taskId);
    
    // 2. 查找下一个任务
    const nextTask = await this.findNextTask(projectId, taskId);
    
    // 3. 派发任务
    if (nextTask) {
      await this.dispatchTask(nextTask);
    }
  }
}

// 优势：
// 1. 没有通知时完全不执行
// 2. 响应延迟 = 网络延迟（毫秒级）
// 3. 代码更简单，逻辑更清晰
```

## 性能对比

### 场景：10 个项目，每个项目 6 个任务，任务平均耗时 2 小时

**定时器轮询（5 秒间隔）：**
```
总轮询次数 = (6 任务 × 2 小时 × 10 项目) / 5 秒
          = 43,200 次 / 12 小时
          = 每小时 3,600 次轮询

每次轮询成本：
- 读取 10 个 project.json
- 检查 60 个任务状态
- 大部分检查都是空的（没有新完成的任务）

CPU 占用：持续低负载（即使没事做也要轮询）
响应延迟：0-5 秒（平均 2.5 秒）
```

**事件驱动：**
```
总处理次数 = 60 次任务完成通知
           = 60 次 / 12 小时
           = 每小时 5 次处理

每次处理成本：
- 读取 1 个 project.json
- 更新进度
- 派发 1 个任务

CPU 占用：零空闲消耗（只在有事时执行）
响应延迟：<100 毫秒
```

**性能提升：**
- 轮询次数减少：**99.86%** (3600 → 5 次/小时)
- 响应延迟降低：**96%** (2.5 秒 → 0.1 秒)
- 资源消耗降低：**>99%**

## 可扩展性对比

### 定时器轮询的瓶颈

```
项目数量增加 → 轮询成本线性增长

10 个项目  → 每小时 3,600 次轮询
100 个项目 → 每小时 36,000 次轮询
1000 个项目 → 每小时 360,000 次轮询（系统崩溃）

解决方案：
- 延长轮询间隔 → 响应延迟增加
- 分布式轮询 → 复杂度爆炸
```

### 事件驱动的扩展

```
项目数量增加 → 处理成本线性增长（但基数很小）

10 个项目  → 每小时 50 次处理
100 个项目 → 每小时 500 次处理
1000 个项目 → 每小时 5000 次处理（轻松应对）

解决方案：
- 天然支持水平扩展
- 多个调度中心实例负载均衡
```

## 可靠性对比

### 定时器轮询的问题

**问题 1：轮询间隔内的任务丢失**
```
T0: 任务 A 完成
T0+2s: 轮询检查（任务 A 还未标记完成）
T0+3s: 任务 A 标记完成
T0+5s: 下次轮询才发现任务 A 完成

延迟：3 秒
```

**问题 2：并发轮询冲突**
```
轮询 1: 读取项目进度（版本 1）
轮询 2: 读取项目进度（版本 1）
轮询 1: 更新进度（版本 2）
轮询 2: 更新进度（版本 2，覆盖了轮询 1 的更新）❌

需要复杂的锁机制
```

### 事件驱动的优势

**优势 1：立即响应**
```
T0: 任务 A 完成
T0+50ms: 提交通知
T0+100ms: 调度中心处理完成

延迟：100 毫秒
```

**优势 2：事件队列天然防冲突**
```
通知 1: 任务 A 完成 → 进入队列
通知 2: 任务 B 完成 → 进入队列
调度中心：按顺序处理通知 1 → 通知 2

无需锁，天然串行
```

## 总结

| 指标 | 定时器轮询 | 事件驱动 | 改进 |
|------|-----------|---------|------|
| 响应延迟 | 2.5 秒（平均） | <0.1 秒 | **25 倍提升** |
| 资源消耗 | 持续轮询 | 按需触发 | **>100 倍优化** |
| 代码复杂度 | 高（定时器 + 锁） | 低（事件处理） | **简化 50%** |
| 可扩展性 | 差（线性增长） | 优（轻松扩展） | **数量级提升** |
| 可靠性 | 中（并发问题） | 高（事件队列） | **显著提升** |

**结论：** 事件驱动在各方面都优于定时器轮询，是现代化的架构选择。
